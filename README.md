# wfm_receiver

Приём, демодуляция и вывод звука широкополосного FM (WFM) через SDR.

Полный тракт работает end-to-end: приём IQ с HackRF или PlutoSDR → канальная
фильтрация и децимация → FM-демодуляция → (опционально) стерео-декодирование
→ de-emphasis → ресемплинг под частоту звуковой карты → воспроизведение через
ALSA. Параллельно с аудио-трактом декодируется RDS (PI/PTY/PS/RadioText),
если станция его передаёт.

## Возможности

- Приём IQ-потока с **HackRF One** и **PlutoSDR (ADALM-PLUTO)** через единый
  интерфейс `hardware::signal_source` (асинхронный колбэк с сырыми байтами).
- Полный DSP-тракт демодуляции WFM: канальный фильтр + децимация
  (equiripple FIR), FM-дискриминатор, de-emphasis, финальный ресемплинг.
- **Стерео** (19 кГц pilot + 38 кГц DSB-SC, с PLL для захвата фазы pilot-тона)
  — флаг `--stereo`; по умолчанию моно.
- **RDS** (57 кГц поднесущая, своя PLL по тому же pilot-тону) — декодирование
  PI, PTY, Program Service (PS) и RadioText (RT); печатается в stdout при
  изменении. Работает всегда, независимо от `--stereo`, т.к. это только
  диагностический текстовый вывод, не влияющий на аудио-тракт.
- Вывод звука в реальном времени через ALSA (48 кГц, mono или stereo).
- Конфигурация источника сигнала из JSON-файла + переопределение параметров
  через аргументы командной строки (Boost.ProgramOptions).
- Архитектура рассчитана на несколько источников сигнала
  (`config::signal_source_type`); сейчас поддержаны HackRF и PlutoSDR.

## Зависимости

- CMake >= 3.16, компилятор с поддержкой C++17
- [libhackrf](https://github.com/greatscottgadgets/hackrf) (`libhackrf-dev`)
- [libiio](https://github.com/analogdevicesinc/libiio) (`libiio-dev`) — для PlutoSDR
- ALSA (`libasound2-dev`) — вывод звука
- Boost >= 1.75, компоненты `json` и `program_options`
  (`libboost-json-dev`, `libboost-program-options-dev`)

На Ubuntu/Debian:

```bash
sudo apt install cmake libhackrf-dev libiio-dev libasound2-dev \
    libboost-json-dev libboost-program-options-dev
```

## Сборка

```bash
cmake -B build -S .
cmake --build build
```

По умолчанию собирается в `Release` (DSP-код в `Debug`/без оптимизаций на
порядок медленнее реального времени — этого достаточно, чтобы приём начал
отставать и звук стал прерывистым). Явно задать другой тип сборки можно
через `-DCMAKE_BUILD_TYPE=...`.

Собираются:
- `libwfm_hw.a` — статическая библиотека (приём с HackRF/PlutoSDR, DSP-тракт,
  аудиовывод, парсинг конфига)
- `wfm_receiver` — исполняемый приёмник

## Запуск

```bash
./build/wfm_receiver --config configs/pluto_signal_source.json --frequency 103400000 --stereo
```

Без `--config` по умолчанию используется `configs/pluto_signal_source.json`.
Перед стартом приложение печатает итоговые параметры (после применения
JSON + CLI-оверрайдов), затем начинает приём и сразу проигрывает звук.
Остановка — `Ctrl+C`.

Если станция передаёт RDS, в stdout появляются строки вида:

```
RDS PI: 0x1234
RDS PS: 'RADIO XYZ'
RDS RT: 'Now playing: some track'
```

PI печатается один раз при первом обнаружении; PS/RT — при каждом изменении
содержимого.

### Аргументы командной строки

| Флаг                 | Описание                                              |
|----------------------|--------------------------------------------------------|
| `-c, --config <path>`| путь к JSON-конфигу (по умолчанию `configs/pluto_signal_source.json`) |
| `-f, --frequency`    | центральная частота, Гц                                |
| `--stereo`           | декодировать стерео (19 кГц pilot + 38 кГц DSB-SC); по умолчанию моно |
| `--lna-gain`         | HackRF: LNA gain, дБ (0-40, шаг 8)                     |
| `--vga-gain`         | HackRF: VGA gain, дБ (0-62, шаг 2)                     |
| `--amp-enable`       | HackRF: включить фронтенд-усилитель (true/false)        |
| `--serial`           | HackRF: серийный номер устройства (если их несколько)   |
| `--gain-mode`        | PlutoSDR: `manual`\|`fast_attack`\|`slow_attack`\|`hybrid` |
| `--rx-gain`          | PlutoSDR: ручной RX gain, дБ (только в режиме `manual`) |
| `--uri`              | PlutoSDR: URI контекста, напр. `usb:5.7.5`, `ip:192.168.2.1` |
| `--audio-device`     | ALSA-устройство воспроизведения (по умолчанию `default`) |
| `-h, --help`         | список аргументов                                      |

Частота дискретизации приёма **не настраивается** — она зафиксирована
(`config::hackrf_sample_rate_hz` = 2 МГц, `config::pluto_sample_rate_hz` ≈
2.083 МГц): это минимум, который поддерживает каждое железо, и одновременно
с запасом хватает на канал WFM (~200 кГц). Фиксация нужна, чтобы
коэффициенты канального фильтра и дециматора не пересчитывались на лету.

У HackRF нет AGC (только фиксированный gain), поэтому по умолчанию
используется максимум (`lna_gain_db=40`, `vga_gain_db=62`, `amp_enable=true`)
— на реальном железе умеренный gain часто не вытягивает станцию из шума. У
PlutoSDR по умолчанию `fast_attack` (AGC) — по той же причине.

### Примеры JSON-конфигов

HackRF (`configs/signal_source.json`):

```json
{
    "source": "hack_rf",
    "frequency_hz": 100000000,
    "lna_gain_db": 40,
    "vga_gain_db": 62,
    "amp_enable": true,
    "serial_number": ""
}
```

PlutoSDR (`configs/pluto_signal_source.json`):

```json
{
    "source": "pluto_sdr",
    "frequency_hz": 100000000,
    "gain_control_mode": "fast_attack",
    "rx_gain_db": 40,
    "uri": ""
}
```

`uri`/`serial_number`, оставленные пустыми, означают автоопределение первого
найденного устройства. Если PlutoSDR виден и по USB, и по сети (`ip://`),
предпочтительнее явно указать `--uri usb:...`: подключение по сети (RNDIS/
iiod) периодически даёт паузы в доставке данных ~200-250 мс, что при
неудачном стечении обстоятельств всё ещё может вызвать редкие прерывания
звука даже с секундным буфером ALSA (см. ниже); по USB таких пауз не было.

## Архитектура приёма и DSP-цепочка

`hardware::signal_source` — абстрактный источник IQ-данных
(`open/start/stop/close`), данные отдаются наружу через колбэк
`iq_data_callback_t` с сырым буфером байт; формат сэмплов (`int8_iq` у
HackRF, `int16_iq` у PlutoSDR) сообщается через `sample_format()`. Фабрика
`hardware::create_signal_source(cfg)` выбирает реализацию по `cfg.type`.

Обработка — цепочка блоков `dsp::block<In, Out>` (`inc/dsp/block.h`):
каждый блок принимает N сэмплов на входе и добавляет результат в выходной
вектор, может быть stateful между вызовами. Полный тракт, как он собран в
`main.cpp`:

```
signal_source (2 МГц / 2.083 МГц, int8/int16)
  -> iq_converter           нормализация в std::complex<float>
  -> equiripple_filter       канальный фильтр + децимация x10 (Parks-McClellan FIR)
  -> quadrature_demodulator  FM-дискриминатор: arg(z[n]*conj(z[n-1]))
  -> [--stereo] stereo_decoder   PLL по 19 кГц pilot + синхронный демод L-R,
                                 выдаёт interleaved L,R на той же частоте
  -> rc_deemphasis_filter    компенсация pre-emphasis (50 мкс), отдельно на
                             канал при стерео
  -> audio_resampler         полифазный ресемплинг под 48 кГц, отдельно на
                             канал при стерео
  -> alsa_sink                воспроизведение (mono или interleaved stereo)

  -> rds_demodulator          (параллельно, всегда) собственная PLL по 19 кГц
                               pilot + 57 кГц поднесущая -> бифазные символы
                               1187.5 бод -> дифференциально декодированные биты
  -> rds_group_decoder         синхронизация блоков по CRC (offset words A/B/C/C'/D),
                               сборка групп -> PI/PTY/PS/RadioText
```

`stereo_decoder` обязательно стоит до `rc_deemphasis_filter`/`audio_resampler`
— их срез уже вырезает 19/38 кГц поднесущие, декодировать стерео после них
невозможно. `rds_demodulator` берёт composite-сигнал в той же точке (сразу
после `quadrature_demodulator`), независимо от `--stereo` — RDS его не
использует, у него собственная PLL по pilot-тону.

Приём и воспроизведение работают на разных потоках, связанных
потокобезопасной очередью (`audio_queue` в `main.cpp`): колбэк приёма
никогда не блокируется на записи в ALSA, чтобы не тормозить сам захват
данных с железа. ALSA-буфер настроен на 1 секунду — с запасом на случай
пауз в доставке данных от источника (см. заметку про `--uri` выше).

### Известные упрощения

- `equiripple_filter` спроектирован ровно под 2 МГц (HackRF); для PlutoSDR
  (2.083334 МГц) частоты среза фильтра сдвинуты на те же ~4%.
- Константы протокола RDS (генератор CRC, offset words A/B/C/C'/D) взяты из
  спецификации IEC 62106; DSP-цепочка (`rds_demodulator`) проверена на
  синтетическом сигнале (100% восстановления бит), а `rds_group_decoder` —
  на синтетических CRC-корректных группах, но полный тракт пока не проверен
  на реальном эфирном сигнале.
- При потере синхронизации блоков (несовпадение хотя бы одной ожидаемой
  группы) `rds_group_decoder` сразу сбрасывает синхронизацию и ищет её заново
  побитово — без допуска на единичные ошибки в шумном канале (сам CRC при
  этом ошибки не исправляет, только детектирует).
- PI и посимвольно PS/RadioText принимаются только после **двух одинаковых**
  подряд идущих приёмов (как в большинстве реальных RDS-декодеров) — это
  фильтрует случайные ложные блок-синхронизации (5 из 1024 возможных
  синдромов совпадают с валидным offset word чисто случайно) и единичные
  испорченные блоки, которые иначе на слабом/шумном сигнале выглядели бы как
  случайный "мусор" в выводе.
- **Найден и исправлен реальный баг pilot-PLL** (общий для `stereo_decoder` и
  `rds_demodulator`): гейны `kp=0.01, ki=0.0001` подобраны на чистом
  синтетическом тоне и давали коэффициент затухания контура ζ≈0.16 при
  реальной амплитуде pilot-тона (~0.1 после bandpass) — контур сильно
  недодемпфирован, поэтому на реальном (шумном) сигнале он не оседает на
  частоте, а непрерывно "звенит": на записи с живого эфира частота PLL
  гуляла ±900 Гц вокруг 19 кГц вместо устойчивого захвата. Подобраны новые
  гейны (`kp=0.00426, ki=0.00000091`, ζ≈0.7, полоса ~10 Гц) — на той же
  записи разброс упал до <1 Гц. На синтетических тестах регрессий нет
  (RDS: 100% бит; stereo: `err_rms` даже немного улучшился).
- Проверено на реальном железе (PlutoSDR, и по сети, и по USB): канал
  приёма/демодуляции и pilot-детектирование работают корректно (19 кГц pilot
  стабильно виден на +25...+40 дБ над шумом), включая целевую проверку по
  реальному списку из 26 станций Санкт-Петербурга (Ретро FM, Авторадио,
  Европа Плюс, DFM и т.д.) — но ни на одной не удалось получить действительно
  декодируемый RDS-сигнал даже после исправления PLL: correlator для
  восстановления битового тайминга показывает совершенно плоское
  распределение энергии (peak/avg ≈ 1.0, т.е. никакой реальной бифазной
  структуры) везде, включая станции с небольшим превышением энергии на
  57 кГц над шумом. Похоже, проверенные станции либо не передают RDS, либо
  соотношение сигнал/шум на поднесущей недостаточно для приёма при текущей
  антенне. Сама DSP-цепочка и протокольный декодер проверены на 100% на
  синтетическом сигнале с известным результатом — положительный кейс
  (реально декодированные PI/PS/RadioText) на эфире пока не подтверждён за
  неимением станции с гарантированно рабочим RDS в зоне приёма.
